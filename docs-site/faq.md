# FAQ

Common questions from judges and new users. A longer judge-focused FAQ lives in the [Full feature guide](../FEATURES.md#9-faq-for-judges).

**Is my money ever controlled by the AI?**
No. The AI only proposes a plan, and four independent layers stand between "the AI said so" and money actually moving. The strategist's own response validator rejects any vault address that isn't already in the allowlisted catalog and any allocation that doesn't sum to roughly 1.0. The AI council's Risk Analyst holds a hard veto — if it returns WITHDRAW with confidence above 0.85, the strategy is discarded outright, no vote-counting needed. The fail-closed eligibility gate drops any protocol with missing, stale, or failing facts (unaudited, thin liquidity, yield paid out that exceeds the protocol's actual revenue) before it can reach you. And you review and can edit every field before approving anything. Past that point, authority to move funds comes entirely from on-chain scope — a deposit-only agent account, pinned to one vault, capped, and expiring. The model's output is inert until you sign, and even then it can't exceed what the contracts allow.

**What exactly am I signing?**
One `funding_router.grant(owner, budget, expiry_ledger, agents[])` transaction. It does two things under a single signature: it sets a SEP-41 token allowance from you to the router — a budget and an expiry ledger enforced by the token contract itself, not by application code — and it deploys one fresh `agent_account` contract per worker, each pre-scoped at deploy time to its own vault, cap, and expiry. Both the allowance and the deploys are authorized by the same source-account credential, which is why one wallet pop-up covers both actions at once.

**How do I pay gas?**
You don't. An own fee-bump relayer (`/api/stellar-relay`) wraps every allowlisted transaction — vault deposit/redeem, router grant/pull, deploys matching a pinned wasm hash — inside a Stellar fee-bump transaction where the relay's own funded keypair pays the network fee. The inner transaction's logic and authorization are untouched; only the fee source changes. Both kill switches still work even if the relayer is down: revoking is a direct, user-paid submission specifically so it doesn't depend on the relay staying up.

**How do I stop everything?**
Two ways, both user-signed and both server-independent: zero the allowance globally with `token.approve(router, 0)`, or revoke a single agent with `agent_account.revoke()`. There's also a narrower option — `owner_withdraw(to)` on one agent redeems its vault shares and sweeps its balance to you immediately, no cooldown, without touching any other agent's scope.

**Is the yield real?**
Yes — deposits supply into a real Blend Capital v2 pool and earn actual testnet lending interest. The vault forwards deposits (as `vfVLT` shares) to a strategy contract that supplies Blend directly; a keeper periodically calls `compound()`, which harvests interest and BLND emissions and sweeps them back in, raising the vault's price-per-share over time. It's testnet, so rates are illustrative and no real funds are involved. See [What's real vs. demo](real-vs-demo.md).

**Which wallets are supported?**
The passkey-based VF Wallet (no seed phrase, optional extension, built-in faucet), plus Freighter, xBull, and Albedo on testnet. VF Wallet's own transaction-building layer runs the same fail-closed eligibility check used in the main strategy flow, so it refuses to even construct a deposit into a vault that fails the safety check — a protection that holds even if you reach the wallet through a path that skipped the strategy UI entirely.

**Which chains does it use?**
Stellar / Soroban is the primary and only required chain. There's an optional cross-chain leg to Base Sepolia via Circle CCTP v2 and ZeroDev session keys, offered in-strategy only while the cross-chain relayer's health probe passes — Base pools are simply absent from the catalog otherwise, not surfaced as a broken option.

**Has it been audited?**
No independent audit yet. This is testnet software. The honest control-and-residual-risk picture is in [SECURITY.md](../SECURITY.md).

**Do I need AI API keys to run it?**
No. Keys are optional — bring a Venice key, set a server-side DeepSeek key, or use neither and rely on the deterministic fallback, which does an equal-split allocation across the static vault catalog: zero cost, and it cannot hang or fail. See [Getting started (developers)](../GETTING_STARTED.md).

**What can a compromised session key actually do?**
Each worker signs with its own fresh, throwaway ed25519 keypair, and the agent account's authorization check enforces the scope pinned at deploy time on every call: one owner, one vault, a per-period spending cap, an expiry, a revoked flag. In the worst case, a leaked session key lets an attacker push only that agent's remaining headroom into the one vault it was already scoped to deposit into — which is where the money was headed anyway. They cannot redirect it to a different vault, cannot touch another agent's funds, and cannot reach your real wallet: the agent's exit path uses a signer separate from the session key and only ever sends funds back to the recorded owner address, never to an arbitrary destination.

**What happens in a market emergency — does it pull my funds automatically?**
Only if you have a live, unexpired mandate at that moment. The lifeboat radar checks pool health roughly every Stellar ledger (about 6 seconds) regardless — utilization spikes, liquidity drops, oracle price divergence — but detecting danger and being *allowed* to act on it are two different things. Without a live mandate, it logs an alarm and does nothing. With one, it calls `emergency_derisk()` to drain every strategy back to vault-idle. The mandate itself is a separate, time-boxed signature (`set_mandate(now + 24h)`) you grant to a dedicated mandate-authority role, distinct from the funding grant, and it has to be renewed periodically or the radar reverts to alarm-only. Resuming afterward needs the same live-mandate check plus a stricter all-clear streak (100 consecutive ledgers, roughly 10 minutes) so the system doesn't flap in and out of the emergency state on a noisy signal.

**Do repeat runs need another signature?**
Not if the on-chain scope still has room. On a repeat run, the orchestrator re-reads each cached agent's scope fresh from the chain — never trusts the local cache — and checks it hasn't expired or been revoked, still has spending headroom, and that the SEP-41 allowance still covers the new run's total. If all of that holds, it skips the grant entirely and funds the run through a relayed `router.pull()`, with zero wallet interaction. The moment any of those conditions fails — expiry passed, allowance exhausted, an agent revoked — it falls back to a fresh grant, which does need your signature again.

**Who pays the gas, really?**
A funded relayer keypair the project controls, held server-side and never exposed to the browser. It isn't a general-purpose sponsor: the relay inspects the specific operation type inside every transaction and only fee-bumps a short allowlist — vault deposit/redeem, router grant/pull, and deploys matching a pinned wasm hash. Anything else, including admin functions, is refused before it ever reaches the relay's signing key. The relay never has custody of funds; it only ever pays the network fee for a transaction someone else already signed and authorized.

**Why Stellar/Soroban instead of an EVM chain?**
The project actually started on an EVM stack — a registry, a depositor, and a 1Shot relay — and migrated fully to Stellar/Soroban, decommissioning the EVM-era stack on 2026-06-21. Soroban's native custom-account authorization lets the agent account define its own spending rules as first-class contract logic, and SEP-41's built-in allowance expiry gives the funding grant a hard cutoff enforced by the token contract itself. Both map directly onto "one scoped signature, many bounded autonomous actions" without needing a bolted-on account-abstraction layer the way EVM chains do — which is also why the optional Base leg needs ZeroDev's ERC-4337 Kernel to get a comparable session-key story, while the Stellar side gets it natively from the protocol.

**Can the team upgrade the contracts under me?**
The funding router can't be upgraded at all — by design, it has no admin and no upgrade function anywhere in its interface. Its constructor pins the agent wasm hash and the funding token permanently at deploy time. The vault is different: its code supports a time-locked upgrade path — an admin-only `schedule_upgrade` starts a 3-day delay before `execute_upgrade` can run, and `cancel_upgrade` can abort it at any point before then — and every step is designed to be publicly surfaced, not just logged internally. Anyone can read the pending state through a `pending_upgrade()` view, each step emits an on-chain event, and the app is wired to show a banner ("Vault upgrade scheduled — executes after [date]. Funds can be withdrawn before then.") the moment one is scheduled, alongside a radar log warning. You'd always have the full delay window to withdraw before a scheduled upgrade takes effect. As of this writing, that upgrade-timelock code lives on the `iq-alter` development branch and isn't merged or deployed yet — the vault wasm currently live predates it, so none of `schedule_upgrade`, `execute_upgrade`, `cancel_upgrade`, or `pending_upgrade` are callable on it today.

**How fast can I exit?**
Immediately, with no cooldown on your own withdrawals — cooldowns in this system gate the *keeper's* rebalancing, never your exit. Redeem vault shares back to USDC any time through the normal position UI, or call `owner_withdraw()` on a single agent to sweep its position out directly. There's also a batched path: the exit router's `sweep(owner, agents, to)` redeems every one of your agents' positions in a single wallet signature instead of one per agent, and it grants no authority of its own — each agent still checks its own recorded owner, so naming an agent you don't own sweeps nothing from it. A vault-level admin pause can block new deposits, but redemption is deliberately never pause-gated, so a pause can never trap funds.
